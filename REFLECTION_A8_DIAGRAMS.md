# REFLECTION_A8_DIAGRAMS.md — Reflection on State and Activity Modeling
## Campus Lost & Found System (CLAFS)

---

## Reflection

Modeling the dynamic behavior of CLAFS through state transition diagrams and activity diagrams was the most technically demanding assignment in this series so far. While previous assignments focused on what the system should do requirements, user stories, use cases this assignment forced me to think about how the system behaves over time, which required a fundamentally different mode of thinking.

**The first challenge was choosing the right level of granularity for states.** When modeling the Item Report object, my initial draft had only three states: Active, Claimed, and Resolved. This felt clean but was misleading it skipped the MatchFound state entirely, which is one of the most valuable features of CLAFS. On the other hand, adding too many states for example, breaking Active into SubActive and SuperActive based on view counts created noise without adding meaningful behavioral information. The rule I settled on was: a state is worth including only if it triggers a different set of allowed transitions or notifications. MatchFound is a valid state because it changes what the system does next it sends notifications and opens a claim window. SubActive would not change anything, so it stayed out.

**The second challenge was modeling parallel actions in activity diagrams.** In the real system, several workflows need to fire multiple actions simultaneously for example, when an admin approves a claim, the system must notify the claimant, notify the reporter, and update the item status, all at once. In Mermaid's flowchart syntax, true parallel (fork/join) notation is limited compared to full UML activity diagram tools. I resolved this by using a subgraph labeled "System Parallel" to indicate concurrent execution, which communicates the intent clearly even if the notation is not strictly UML parallel fork syntax. This was a pragmatic decision driven by tooling constraints rather than a modeling error.

**The third challenge was aligning the diagrams with Agile user stories from Assignment 6.** User stories are written from the user's perspective "As a student, I want to submit a claim..." whereas activity diagrams model the system's perspective, including all the validation and decision logic that happens behind the scenes. Mapping a single user story to a workflow sometimes felt like a mismatch in scope: US-006 (Submit Claim) maps to an activity diagram with 12+ steps, most of which the user never sees. I resolved this by treating user stories as the motivation for a workflow rather than a one-to-one mapping the user story describes the desired outcome, the activity diagram describes how the system achieves it.

**Comparing state diagrams and activity diagrams** reveals that they answer fundamentally different questions. State diagrams answer: "What condition is this object in, and what can happen to it next?" They are object-centric and focus on lifecycle. Activity diagrams answer: "What steps does this process follow, and who is responsible for each?" They are process-centric and focus on flow. For CLAFS, both are necessary the Item Report state diagram shows that a Resolved item cannot be claimed again, while the Report Lost Item activity diagram shows exactly how the system gets an item from Draft to Active. Together they provide a complete behavioral picture that neither could provide alone.

The most valuable lesson from this assignment is that dynamic modeling exposes assumptions that static modeling hides. My requirements and use cases said "the system shall notify users" but it was only when drawing the notification activity diagram that I had to decide: what if the user has notifications disabled? What if the delivery fails? What happens after three retries? These edge cases were invisible in the requirements and only surfaced through the act of modeling the workflow step by step.
