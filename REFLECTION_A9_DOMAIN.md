# REFLECTION_A9_DOMAIN.md — Reflection on Domain Modeling and Class Diagram Design
## Campus Lost & Found System (CLAFS)

---

## Reflection

Designing the domain model and class diagram for CLAFS was the assignment that brought together everything from the previous eight assignments into a single, coherent structural picture. It was also the most technically demanding from an object-oriented design perspective, because every decision — which attributes belong where, which relationships are composition versus aggregation, whether to use inheritance or not — has downstream consequences for how the system will actually be implemented.

**The first challenge was abstraction — deciding what deserves to be a class.** My initial instinct was to model everything I could think of: categories, campuses, file types, notification templates. I quickly realised this was over-engineering. A class in a domain model should represent something that has both data (attributes) and behaviour (methods) and that the system needs to track over time. File types and categories are simple string values, not entities. Notification templates are configuration, not domain objects. Applying this filter reduced my initial list of fifteen potential classes down to the seven core entities that genuinely drive the CLAFS domain. The lesson here was that good abstraction means knowing what to leave out, not just what to include.

**The second challenge was choosing between inheritance, composition, and aggregation.** The most difficult decision was how to model the relationship between `User` and its subtypes `Admin`, `Student`, and `Lecturer`. My first draft used a single `User` class with a `role` attribute and no inheritance at all, on the grounds that all three share the same authentication mechanism. This was simpler but less expressive — it meant the class diagram did not show that `Admin` has methods like `manageItemReports()` and `exportReport()` that `Student` and `Lecturer` do not have. I switched to inheritance to make the role-specific behaviour visible in the diagram, even though at the database level a single users table with a role column is the more practical implementation. This is a classic trade-off between design clarity and implementation pragmatism — I chose clarity for the purposes of this model.

The distinction between composition and aggregation was also non-trivial. `UserSession` is a composition with `User` because a session is entirely dependent on the user existing — it has no independent identity. `Claim`, however, is an aggregation with `ItemReport` because a claim record has meaning outside of its parent item — an admin might query all claims by a specific user across multiple items. Getting this distinction right matters because it affects cascade behaviour: if an `ItemReport` is deleted, its `Image` and `AdminCase` should be deleted too (composition), but its `Claims` and `Notifications` might need to be retained for audit purposes (aggregation).

**The third challenge was aligning the class diagram with state diagrams from Assignment 8.** Each state modelled in the state transition diagrams had to be reflected as an attribute in the class diagram. The `ItemReport` status (ACTIVE, RESOLVED, EXPIRED), the `Claim` status (PENDING, UNDER_REVIEW, APPROVED, REJECTED), the `Notification` delivery status (QUEUED, SENT, FAILED, ABANDONED), and the `UserSession` status (ACTIVE, IDLE, EXPIRED, TERMINATED) all needed to appear as `status` attributes in their respective classes. This cross-checking process was valuable because it revealed one gap — the `AdminCase` class had states in Assignment 8 (OPEN, ESCALATED, PENDING_COLLECTION) that I had not initially included as a status attribute in the class. Catching this kind of inconsistency is exactly what the traceability process is designed to do.

**The most important lesson from this assignment** is that a class diagram is not just a technical artifact — it is a communication tool. Every naming decision, every multiplicity label, and every relationship type conveys information to whoever reads it, whether that is a future developer, a project reviewer, or myself returning to the project after a break. The discipline of making every decision explicit — rather than leaving it implied or ambiguous — is what separates a professional design model from a rough sketch, and it is a discipline I intend to carry forward into the implementation phase of CLAFS.
