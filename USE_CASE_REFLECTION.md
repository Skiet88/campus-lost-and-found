# REFLECTION5.md — Challenges in Translating Requirements to Use Cases and Tests
## Campus Lost & Found System (CLAFS)

---

Translating the stakeholder concerns and system requirements documented in Assignment 4 into use case diagrams, specifications, and test cases was a more nuanced process than I initially anticipated. While the requirements gave a clear picture of *what* the system needed to do, converting them into structured use cases and testable scenarios forced me to think at a much deeper level about *how* the system would behave in every situation, including the ones that go wrong.

**The first challenge I encountered was defining the right level of granularity for use cases.** When I began drafting the use case diagram, I initially created too many use cases — almost one for every functional requirement. This made the diagram cluttered and hard to follow. I had to step back and think about what constitutes a meaningful interaction between an actor and the system, as opposed to what is simply an internal system behaviour. For example, "Upload Item Image" is not something a user consciously initiates on its own — it is always part of reporting a lost or found item. Recognising this allowed me to correctly model it as an *include* relationship rather than a standalone use case, which cleaned up the diagram significantly.

**The second challenge was modelling actor relationships accurately.** The system has several user types — Student, Lecturer/Staff, New Student, Admin, IT Department, and University Management — and deciding how to represent their relationships was not straightforward. I initially treated each as a fully independent actor with their own use cases. However, on reflection, Lecturers and New Students share all the same system capabilities as regular Students, just with different contexts. Using UML generalisation to show that Lecturer and New Student are specialisations of Student was a cleaner and more accurate representation, and it also reflected the role-based access control design from Assignment 3 more faithfully.

**The third challenge arose when writing the alternative flows in the use case specifications.** It is easy to write the "happy path" — the basic flow where everything goes correctly. The alternative flows, however, require thinking adversarially: what if the image is too large? What if the user loses their internet connection mid-upload? What if an admin tries to resolve a case that has no approved claim? Writing these edge cases was time-consuming but valuable, because it surfaced gaps in the requirements that were not explicitly covered in Assignment 4. For instance, the scenario where an admin accidentally marks an item as resolved led me to add a 24-hour undo window, which was not in the original functional requirements but clearly serves user needs.

**The fourth challenge was designing test cases that are specific and reproducible without being tied to implementation details.** A common mistake in test case writing is to reference specific UI elements or internal variables that might change as the system is built. I had to write test steps at a level of abstraction that describes *what the user does* rather than *which button they click*, making the tests durable across different stages of development. This was particularly difficult for the non-functional tests, where simulating 500 concurrent users or injecting malicious SQL strings required thinking about external tools like Apache JMeter and OWASP ZAP rather than just the application itself.

**Finally, maintaining traceability throughout was a discipline in itself.** Every use case needed to link back to at least one functional requirement from Assignment 4, and every test case needed to reference a requirement ID. This traceability matrix approach, while slightly tedious, made me realise how important it is in real software engineering practice — it is the mechanism that ensures nothing gets built that was not asked for, and nothing that was asked for gets forgotten.

Overall, this assignment deepened my appreciation for the requirements engineering process. The gap between a written requirement and a working, tested system feature is bridged by exactly this kind of careful modelling work.
